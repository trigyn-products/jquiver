# Performance Guidelines

## Purpose
Capture safe v1 performance guidance for JQuiver metadata-driven modules.

## When to use this file
Use this file when reviewing grids, dashboards, dashlets, Dynamic REST APIs, additional datasource queries, templates, or public pages.

## Known facts
- Grids and dashlets may execute SQL against the main schema or additional datasources.
- Large listing pages should be designed with pagination and filters.
- Known frontend context includes pqGrid.

## Performance risks
- Grid queries over large tables without indexes.
- Dashlets executing expensive aggregates.
- Autocomplete queries without search limits.
- Dynamic REST APIs returning large payloads.
- Views with complex joins and no filtering.
- Cross-database datasource latency.
- File download endpoints serving large files.
- Public pages loading too many assets or rows.

## Guidance
- Prefer views for complex listing joins, but verify view performance.
- Use filters and pagination for grids.
- Limit autocomplete result size.
- Add indexes on common filter/sort columns where appropriate.
- Keep dashboard queries aggregate and bounded.
- Avoid N+1 service/API calls from templates.
- Cache only when source behavior confirms cache invalidation rules.

## Safe AI-agent usage
- Do not propose indexes without checking existing indexes and query plans.
- Do not assume caching exists.
- Do not run heavy diagnostics on production without approval.
- Include rollback considerations for index or view changes.
